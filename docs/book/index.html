A minimal, fast, S3-compatible blob storage for the JVM.

Минималистичное, быстрое, S3-совместимое хранилище блобов для JVM.

1. What is Boins1. Что такое Boins

Boins stores binary objects (blobs) by packing them together into a few large files instead of creating one file per object. A compact fixed-size index maps a numeric blob id to the exact byte range, so reading a blob costs a single positional read — no directory walks, no per-object file handles, no filesystem metadata pressure at hundreds of millions of objects.

Boins runs in two modes:

  • Embedded — the boins-core library inside your JVM process, zero dependencies;
  • Server — boins-server exposes the storage over an S3-compatible HTTP API, so any AWS SDK, aws s3 CLI, MinIO client or existing S3 integration works out of the box, including presigned URLs for direct-to-storage transfers.

Design values, in order: minimalism, reliability, performance, scalability.

Boins хранит бинарные объекты (блобы), упаковывая их вместе в несколько больших файлов, вместо того чтобы создавать по файлу на объект. Компактный индекс с записями фиксированного размера отображает числовой идентификатор блоба в точный диапазон байтов, поэтому чтение — это одно позиционное чтение с диска: без обхода каталогов, без файловых дескрипторов на каждый объект, без нагрузки на метаданные ФС даже при сотнях миллионов объектов.

Boins работает в двух режимах:

  • Embedded — библиотека boins-core внутри вашего JVM-процесса, ноль зависимостей;
  • Сервер — boins-server открывает хранилище через S3-совместимый HTTP API: работает любой AWS SDK, CLI aws s3, клиент MinIO и существующие интеграции с S3, включая подписанные ссылки (presigned URL) для передачи данных напрямую в хранилище.

Ценности проекта по порядку: минимализм, надёжность, производительность, масштабируемость.

2.1 The index record (80 bytes)2.1 Индексная запись (80 байт)

blobFileIndex:2blobPosition:8blobSize:8cellSize:8 createTime:8deleteTime:8keyPosition:8metaPosition:8 contentTypeId:2md5:16partCount:4

Everything S3 needs lives in the record: the MD5 digest makes ETags instant, partCount distinguishes multipart etags, keyPosition/metaPosition point into the append-only string heap holding the object key and user metadata, contentTypeId references a tiny in-memory dictionary. deleteTime = -1 means the blob is alive; a deleted record keeps its cellSize so the region can be reused.

В записи есть всё, что нужно S3: дайджест MD5 даёт мгновенные ETag, partCount отличает multipart-etag, keyPosition/metaPosition указывают в append-only кучу строк с ключом объекта и пользовательскими метаданными, contentTypeId ссылается на маленький словарь в памяти. deleteTime = -1 — блоб жив; удалённая запись сохраняет cellSize, чтобы регион можно было переиспользовать.

2.2 Writes2.2 Запись

A write first asks the free-cell registry for a deleted cell of a fitting size (best fit). If none fits, the blob is appended: the writer reserves a region with an atomic counter and streams content directly into it — on SSD many writers stream concurrently into disjoint regions; on HDD disk writes are serialized per chunk to keep the access pattern mostly sequential without letting one slow client stall the others. The MD5 digest is computed during the copy. Only after the content is fully in place is the index record inserted — the record is the commit point.

If a client aborts mid-upload, the reserved region is registered as a free cell and a later write of similar size reclaims it.

Запись сначала спрашивает у реестра свободных ячеек подходящую по размеру ячейку удалённого блоба (best fit). Если такой нет — блоб дописывается в конец: писатель резервирует регион атомарным счётчиком и стримит содержимое прямо в него. На SSD несколько писателей пишут одновременно в непересекающиеся регионы; на HDD запись сериализуется по чанкам — паттерн доступа остаётся почти последовательным, но медленный клиент не блокирует остальных. MD5 считается во время копирования. Индексная запись вставляется только когда содержимое полностью на месте — запись в индексе и есть точка фиксации.

Если клиент оборвал загрузку, зарезервированный регион регистрируется как свободная ячейка, и последующая запись похожего размера использует его.

2.3 Free cells and the waste tolerance2.3 Свободные ячейки и допуск потерь

Blobs in Boins can be deleted. The cell a deleted blob occupied becomes free, and a new blob may take such a cell of equal or larger size. A larger cell, though, may be taken only if the wasted fraction is acceptable for that cell size: maxWaste(s) = 1 / e1.7·(log₂s/10)^0.7. Small cells tolerate generous waste; large cells demand a tight fit, which keeps absolute waste bounded:

В Boins блобы можно удалять. Ячейка, которую занимал удалённый блоб, считается освободившейся. Новый блоб может занять такую ячейку равного или большего размера. Причём занять ячейку большего размера можно, только если доля потерь приемлема для этого размера ячейки: maxWaste(s) = 1 / e1.7·(log₂s/10)^0.7. Маленькие ячейки терпят щедрые потери, большие требуют плотной посадки — абсолютные потери остаются ограниченными:

2.4 Crash consistency2.4 Устойчивость к сбоям

Boins deliberately avoids a write-ahead log. Instead, the free-cell registry is allowed to be stale: on startup every registry entry is cross-checked against the index and dropped if the donor record is alive, missing, or its cell was already consumed by a replacement blob (replacement zeroes the donor's cellSize in the same index fsync as the new insert). A torn tail in a bucket's key index is detected by per-record CRC32 and truncated. The result: after a crash the storage self-heals on open, and the worst case is a bounded amount of leaked space — never double-allocated cells or corrupted reads.

Boins сознательно обходится без write-ahead-лога. Вместо этого реестру свободных ячеек разрешено «отставать»: при старте каждая его запись сверяется с индексом и отбрасывается, если донорская запись жива, отсутствует или её ячейка уже занята блобом-заменой (замена обнуляет cellSize донора тем же fsync индекса, что и вставка новой записи). Оборванный хвост индекса ключей бакета определяется по CRC32 каждой записи и усекается. Итог: после сбоя хранилище самовосстанавливается при открытии; худший случай — ограниченная утечка места, но никогда — двойное выделение ячейки или повреждённое чтение.

2.6 Scaling with repositories2.6 Масштабирование репозиториями

A storage may span several repositories (disks). Each owns a contiguous blob-id range starting at its blobIdOffset; reads route by id with a floor lookup. Inserts pick a repository randomly, weighted by free disk space, so disks fill evenly. A repository manifest pins the format version and offset, making repositories self-describing and safe to relocate.

Хранилище может состоять из нескольких репозиториев (дисков). Каждый владеет непрерывным диапазоном идентификаторов, начиная со своего blobIdOffset; чтение маршрутизируется по id. Вставка выбирает репозиторий случайно, с весом по свободному месту, поэтому диски заполняются равномерно. Манифест репозитория фиксирует версию формата и offset — репозиторий самоописываем и его безопасно переносить.

Beyond one machine. Boins scales horizontally as a fleet of independent servers — there is no cluster protocol to operate, no consensus to babysit. Sharding happens at the bucket level: every bucket lives wholly on one host, and applications keep a small bucket → endpoint map. Presigned URLs carry the host name, so direct-to-storage transfers land on the right server automatically. Whole buckets move between servers with standard S3 tooling (aws s3 sync, rclone) — blob ids are internal and never leave a host.

One planning rule pays for itself later: give every server its own region of blobIdOffset values (server A: 0…, server B: 2·10¹²…, server C: 4·10¹²…). Ids then never collide across the fleet, which keeps the cheap file-level consolidation path open when you need to merge servers. The operational playbooks — growing a server, adding servers, shrinking back to one, moving hosts — are recipes 12.7–12.11.

За пределами одной машины. Boins масштабируется горизонтально как парк независимых серверов — нет кластерного протокола, нет консенсуса, за которым надо следить. Шардирование идёт на уровне бакетов: каждый бакет целиком живёт на одном хосте, а приложение хранит маленькую карту «бакет → endpoint». Presigned-ссылки содержат имя хоста, поэтому передачи напрямую в хранилище автоматически попадают на нужный сервер. Целые бакеты переносятся между серверами штатными S3-инструментами (aws s3 sync, rclone) — идентификаторы блобов внутренние и хост не покидают.

Одно правило планирования окупается позже: выделяйте каждому серверу собственную область значений blobIdOffset (сервер A: 0…, сервер B: 2·10¹²…, сервер C: 4·10¹²…). Тогда идентификаторы не пересекаются во всём парке, и остаётся открытым дешёвый путь файловой консолидации, когда серверы понадобится слить. Операционные сценарии — расширение сервера, добавление серверов, сжатие обратно до одного, переезд хоста — это рецепты 12.7–12.11.

2.7 Thread safety2.7 Потокобезопасность

Boins is fully thread-safe — in embedded mode too. Open one instance per storage and share it across any number of threads (platform or virtual): every public method of Boins may be called concurrently without external synchronization. The server mode builds on exactly this property — Javalin serves each request on its own virtual thread against the single shared core.

  • Writes reserve disjoint regions with atomic counters; on SSD any number of writers stream concurrently, on HDD disk writes are serialized per chunk. Index appends take a short per-repository lock (an 80-byte write).
  • Reads are positional and lock-free — unlimited concurrent readers, even of the same blob, and readers never block writers.
  • Deletes are guarded by striped per-blob-id locks, so a concurrent double delete cannot hand the same cell to two writers.
  • Metrics use LongAdders and lock-free windows — negligible hot-path cost.

The one rule: a storage belongs to a single Boins instance in a single process. Do not open the same repository directories from two instances or two JVMs — there is no cross-process file lock, and concurrent processes would corrupt the storage. If several applications need the same data, put boins-server in front of it.

Boins полностью потокобезопасен — в том числе в режиме embedded. Откройте один экземпляр на хранилище и делите его между любым числом потоков (платформенных или виртуальных): каждый публичный метод Boins можно вызывать конкурентно без внешней синхронизации. Серверный режим построен ровно на этом свойстве — Javalin обслуживает каждый запрос на своём виртуальном потоке поверх единственного общего ядра.

  • Запись резервирует непересекающиеся регионы атомарными счётчиками; на SSD писатели стримят одновременно, на HDD запись на диск сериализуется по чанкам. Вставка в индекс берёт короткий лок на репозиторий (запись 80 байт).
  • Чтение позиционное и без блокировок — сколько угодно одновременных читателей, даже одного и того же блоба; читатели никогда не блокируют писателей.
  • Удаление защищено полосатыми (striped) локами по blob id — конкурентное двойное удаление не отдаст одну ячейку двум писателям.
  • Метрики — LongAdder-ы и lock-free окна, издержки на горячем пути ничтожны.

Единственное правило: хранилище принадлежит одному экземпляру Boins в одном процессе. Не открывайте одни и те же каталоги репозиториев из двух экземпляров или двух JVM — межпроцессной файловой блокировки нет, и параллельные процессы повредят хранилище. Если одни данные нужны нескольким приложениям — поставьте перед ними boins-server.

3. Embedded mode3. Режим embedded

The core is a single dependency-free library:

Ядро — одна библиотека без зависимостей:

BoinsOptions options = BoinsOptions.builder()
        .addRepository(Path.of("/data/repo1"), 0L, DiskType.SSD)
        .addRepository(Path.of("/mnt/disk2/repo2"), 1_000_000_000_000L, DiskType.HDD)
        .blobFileLimit(36)                    // 2^36 = 64 GiB per blob file
        .fsyncMode(FsyncMode.INTERVAL)
        .build();

try (Boins boins = Boins.open(options)) {
    WriteResult w = boins.write(Path.of("report.pdf"),
            new BlobMetadata("docs/report.pdf", "application/pdf"));

    BlobInfo info = boins.info(w.blobId());   // size, etag, key, metadata...

    try (InputStream in = boins.read(w.blobId())) { /* stream it */ }
    try (InputStream in = boins.read(w.blobId(), 0, 1024)) { /* range read */ }

    boins.delete(w.blobId());                 // cell becomes reusable
}

Streaming writes (write(InputStream, length, metadata)) copy straight from the source into the blob file — no temporary files, MD5 on the fly.

The storage can also be configured by a YAML file instead of the builder. If the file does not exist, Boins creates it with commented defaults (parent directories included); an existing but invalid file fails with a clear BoinsException. Relative paths inside the file are resolved against the file's directory:

Потоковая запись (write(InputStream, length, metadata)) копирует данные из источника прямо в blob-файл — без временных файлов, MD5 считается на лету.

Хранилище можно конфигурировать и YAML-файлом вместо билдера. Если файла нет, Boins создаст его с закомментированными значениями по умолчанию (включая родительские каталоги); существующий, но невалидный файл даёт понятный BoinsException. Относительные пути внутри файла разрешаются от каталога самого файла:

// Creates ./boins/boins-options.yaml with defaults on first run
try (Boins boins = Boins.open(Path.of("./boins/boins-options.yaml"))) {
    boins.write(data, BlobMetadata.ofKey("first-blob"));
}

Созданный файл выглядит так — каждое поле снабжено комментарием, и ровно этот формат принимает обратно строгий парсер:

Самостоятельный сервер запускается с YAML-конфигом:

Встраиваемый сервер (тот же API без YAML): соберите BoinsServerConfig в коде и вызовите BoinsServer.start(config); port: 0 займёт эфемерный порт.

Бакеты определяются конфигурацией, а не вызовом S3 CreateBucket — учётным данным место в конфиге, и публичная точка не должна создавать бакеты сама.

7. Fault store7. Хранилище исключений

Exceptional events are saved as flat text files in one directory — no WAL, no trees. Each unique fault gets one report file named <Type>-<hash>.txt. Uniqueness is the exception type plus the first N stack trace lines (default 5); repeated occurrences bump a counter, and the report file is rewritten at most once per cooldown (default 60 s) — a fault storm cannot flood the disk. A file cap (default 1000) evicts the least recently seen fault. HTTP request context — method, path and all headers — is stored with server-side errors.

Исключительные события сохраняются плоскими текстовыми файлами в одной папке — без WAL и без деревьев. Каждому уникальному исключению — один файл-отчёт <Тип>-<hash>.txt. Уникальность — тип исключения плюс первые N строк стектрейса (по умолчанию 5); повторы увеличивают счётчик, а файл перезаписывается не чаще одного раза за cooldown (по умолчанию 60 с) — шторм исключений не зальёт диск. Лимит файлов (по умолчанию 1000) вытесняет самое давнее исключение. С серверными ошибками сохраняется HTTP-контекст запроса: метод, путь и все заголовки.

8. Metrics8. Метрики

The core collects traffic counters (bytes/blobs written, read, deleted, errors), sliding-window rates over the last 10 seconds (bytes/s and ops/s for reads and writes) and average speeds over active I/O time. Counters are persisted to a properties file on the storage file pool every 10 s (configurable) and on shutdown, then loaded back on open — totals accumulate across restarts. The server adds HTTP counters per S3 operation. Everything is visible at GET /_admin/metrics.

Ядро собирает счётчики трафика (байты/блобы записанные, прочитанные, удалённые, ошибки), скорости в скользящем окне последних 10 секунд (байт/с и оп/с для чтения и записи) и средние скорости за активное время ввода-вывода. 12.7 Growing a server: a new disk, a new repository12.7 Расширение сервера: новый диск, новый репозиторий

The disk is filling up — mount another one and add a repository to the storage section. Two rules: never change the offset of an existing repository, and give the new one an offset far above every id that could already exist (steps of 10¹² leave room for ~10¹² blobs per repository):

Диск заполняется — подключите ещё один и добавьте репозиторий в секцию storage. Два правила: никогда не меняйте offset существующего репозитория и дайте новому offset заведомо выше всех уже возможных id (шаг 10¹² оставляет место под ~10¹² блобов на репозиторий):

storage:
  repositories:
    - dir: /data/nvme1/repo1
      blobIdOffset: 0              # existing repository — untouched
      diskType: SSD
    - dir: /data/nvme2/repo2       # the new disk
      blobIdOffset: 1000000000000  # 10^12 — far above any existing id
      diskType: SSD

Restart the server (graceful — a systemctl restart is enough). Existing blobs stay exactly where they are; new inserts immediately start balancing across both disks, weighted by free space.

Moving a repository to a bigger disk on the same server works the same way: stop the server, move the directory (mv /data/old/repo1 /data/new/repo1), update its dir: in the config, start. The manifest inside the directory carries the offset, so a path change is safe — and a wrong directory is rejected at startup.

Перезапустите сервер (мягко — достаточно systemctl restart). Существующие блобы остаются на своих местах; новые вставки сразу начинают распределяться по обоим дискам с весом по свободному месту.

Перенос репозитория на диск побольше в рамках того же сервера делается так же: остановите сервер, перенесите каталог (mv /data/old/repo1 /data/new/repo1), поправьте его dir: в конфиге, запустите. Манифест внутри каталога хранит offset, поэтому смена пути безопасна — а «не тот» каталог будет отвергнут при старте.

12.8 Scaling out: from one server to a fleet12.8 Распределение: от одного сервера к парку

The startup grew and one machine is not enough. Boins servers are independent — scaling out is an application-level pattern, not a cluster to operate:

  1. Shard by bucket. A bucket lives wholly on one host. Split by data kind (user-media → s3-a, analytics → s3-b) or by tenant ranges. Keep a bucket → endpoint map in your app config; presigned URLs carry the host automatically.
  2. Bring up the new server (section 11) and, if a bucket must move, transfer it with standard tooling — then flip the endpoint map:

Стартап вырос, и одной машины мало. Серверы Boins независимы — распределение является паттерном уровня приложения, а не кластером, который нужно администрировать:

  1. Шардируйте по бакетам. Бакет целиком живёт на одном хосте. Делите по типу данных (user-media → s3-a, analytics → s3-b) или по диапазонам тенантов. Держите карту «бакет → endpoint» в конфиге приложения; presigned-ссылки сами содержат хост.
  2. Поднимите новый сервер (раздел 11) и, если бакет надо переместить, перенесите его штатным инструментом — затем переключите карту endpoint-ов:
# move a whole bucket between servers (ids are internal — S3 level is the way).
# aws s3 sync uses one endpoint per command, so stage through a directory:
aws s3 sync s3://user-media ./migrate-user-media --endpoint-url http://s3-a.internal:9000
aws s3 sync ./migrate-user-media s3://user-media --endpoint-url http://s3-b.internal:9000

# or copy directly between two remotes with rclone (no staging, resumable):
rclone sync boins-a:user-media boins-b:user-media --checksum

Plan for a future merge on day one: give each server its own blobIdOffset region — server A starts at 0, server B at 2·10¹², server C at 4·10¹². Ids will never collide across the fleet, which keeps the fast file-level consolidation (12.9) available. And if you put nginx or a load balancer in front, preserve the Host header — SigV4 signs it:

Планируйте будущее слияние с первого дня: выделите каждому серверу свою область blobIdOffset — сервер A начинается с 0, сервер B с 2·10¹², сервер C с 4·10¹². Идентификаторы никогда не пересекутся в парке, и быстрая файловая консолидация (12.9) останется доступной. А если перед сервером стоит nginx или балансировщик — сохраняйте заголовок Host: он входит в подпись SigV4:

# nginx in front of Boins
location / {
    proxy_pass http://127.0.0.1:9000;
    proxy_set_header Host $host;      # SigV4 signs the Host header
    proxy_request_buffering off;      # stream uploads, don't spool to disk
    client_max_body_size 0;           # Boins enforces its own limits
}

12.9 Consolidating: many servers back into one12.9 Консолидация: несколько серверов снова в один

The opposite case: capacity was planned for high load, the load never came, and one machine is plenty. Two paths, from universal to fastest:

Path 1 — S3-level sync (always works). Define all the buckets on the target server, copy each bucket with aws s3 sync/rclone from its old host, switch the endpoint map, decommission the old servers. Blob ids are reassigned and the storage is repacked densely — the consolidated server starts clean. Keep the old servers read-only during the copy (easiest: point all writers at the new endpoint first).

Path 2 — file-level adoption (no re-upload). Possible when bucket names are unique across servers and their repositories use non-overlapping offset regions (the 12.8 planning rule). Then the target server simply adopts the files:

Обратный случай: ёмкость закладывали под высокие нагрузки, нагрузка не пришла, и одной машины достаточно. Два пути — от универсального к самому быстрому:

Путь 1 — синхронизация на уровне S3 (работает всегда). Опишите все бакеты на целевом сервере, перелейте каждый бакет aws s3 sync/rclone с его старого хоста, переключите карту endpoint-ов, выведите старые серверы из эксплуатации. Идентификаторы блобов назначаются заново, хранилище пересобирается плотно — консолидированный сервер начинает «с чистого листа». На время копирования держите старые серверы в режиме «только чтение» (проще всего сначала перевести всех писателей на новый endpoint).

Путь 2 — файловое принятие (без перезаливки). Возможен, когда имена бакетов уникальны между серверами и их репозитории используют непересекающиеся области offset-ов (правило планирования из 12.8). Тогда целевой сервер просто принимает файлы:

# 1. stop every old server (graceful)         systemctl stop boins
# 2. copy all repository dirs to the target host
# 3. list them ALL in the target storage config with their ORIGINAL offsets:
storage:
  repositories:
    - { dir: /data/a-repo1, blobIdOffset: 0 }              # from server A
    - { dir: /data/a-repo2, blobIdOffset: 1000000000000 }  # from server A
    - { dir: /data/b-repo1, blobIdOffset: 2000000000000 }  # from server B
# 4. copy each bucket dir (keys.boins) from the old servers
#    into the target bucketDataDir, and declare the buckets in the config
# 5. start; check /_admin/state — blob counts must match the old servers
Path 2 caveats: manifests verify every adopted repository (a wrong offset is rejected at startup). Keep only the target's free-cells.boins and metrics file — the old servers' registries are discarded, which merely leaks their reusable cells (bounded, observable in /_admin/state). If offsets collide between servers, path 2 is impossible — use path 1. Оговорки пути 2: манифесты проверяют каждый принятый репозиторий («не тот» offset отвергается при старте). Оставьте только free-cells.boins и файл метрик целевого сервера — реестры старых серверов отбрасываются, что лишь теряет их переиспользуемые ячейки (ограниченно, видно в /_admin/state). Если offset-ы серверов пересекаются, путь 2 невозможен — используйте путь 1.

12.10 Moving a whole server to a new host12.10 Переезд сервера на новый хост целиком

  1. systemctl stop boins on the old host — the graceful shutdown drains requests and flushes every file.
  2. Copy the data and the config: /var/lib/boins (repositories, buckets, faults) and /etc/boins/boins.yaml; install the distribution on the new host (section 11).
  3. Same paths → zero config edits. Different paths → update dir:, bucketDataDir, faults.dir; repositories are relocatable, the manifest carries their identity.
  4. Start, then compare /_admin/state (blob counts, max ids) with the old host before repointing DNS or the endpoint map.
  1. systemctl stop boins на старом хосте — graceful shutdown дообслужит запросы и сбросит все файлы на диск.
  2. Скопируйте данные и конфиг: /var/lib/boins (репозитории, бакеты, faults) и /etc/boins/boins.yaml; установите дистрибутив на новом хосте (раздел 11).
  3. Пути совпадают → конфиг не трогаем. Пути другие → обновите dir:, bucketDataDir, faults.dir; репозитории переносимы, манифест несёт их идентичность.
  4. Запустите и сравните /_admin/state (число блобов, максимальные id) со старым хостом, прежде чем переключать DNS или карту endpoint-ов.

12.11 Server selection and bucket planning12.11 Выбор сервера и планирование бакетов

With several Boins servers, a read must land on the host that has the object — so the location of every blob must be computable, never searched for. The routing key is (bucket, key): blob ids are internal and never leave a host. Three schemes, in order of growth:

  1. Bucket → endpoint map in the application config (user-media → s3-a). Simplest; enough while every bucket fits its server.
  2. Entity sharding. Shard by a stable entity — user id, tenant id — not by object: shard = hash(userId) % N → bucket media-3 → its endpoint. Reads recompute the same hash. An entity's files stay together, prefix listing keeps working, and rebalancing moves entities, not random objects.
  3. A directory in your database. Choose the server at upload time and store (key, endpoint) in the row you already keep for the file. Reads hit that row anyway, so the endpoint is free; rebalancing is copy + update the row.

When assigning a server to a new entity, pick it like Boins picks repositories: randomly, weighted by free space (from /_admin/state, cached for minutes). Presigned URLs work with every scheme — the backend embeds the right host when signing.

Planning so a bucket never outgrows a server:

  • If a data kind can grow unboundedly, shard it from day one: create media-0…media-7 even if all eight start on one server. Splitting later is then "move whole buckets" (trivial, recipe 12.8) instead of "split one bucket" (migration below).
  • Keys always carry an entity prefix (users/{id}/…) — this is what makes lazy migration and per-entity moves possible at all.
  • Watch /_admin/state and act at ~70 % disk usage: recipe 12.7 (add a disk) buys time cheaply.

The bucket did outgrow its server — the split procedure:

  1. Buy time: add a repository on a new disk (12.7).
  2. Create shard buckets on two (or more) servers, e.g. media-0 on A, media-1 on B. Route new writes by entity hash immediately.
  3. Migrate old entities in the background, prefix by prefix: aws s3 sync/rclone of users/{id}/… to the entity's new shard.
  4. During migration read with a fallback: try the new location, on 404 read the old bucket (or track a per-entity "migrated" flag in the database — cleaner).
  5. When the sync is complete, drop the fallback and delete the old bucket from the config.

If keys have no entity prefix and hashing is impossible, use scheme 3: point new uploads at the new bucket via the database row, leave old rows on the old server, and migrate lazily or not at all.

Когда серверов Boins несколько, чтение обязано попасть на хост, где объект лежит, — значит, расположение каждого блоба должно вычисляться, а не разыскиваться. Ключ маршрутизации — (bucket, key): идентификаторы блобов внутренние и хост не покидают. Три схемы, по мере роста:

  1. Карта «бакет → endpoint» в конфиге приложения (user-media → s3-a). Проще всего; достаточно, пока каждый бакет помещается на свой сервер.
  2. Шардирование по сущности. Делите по стабильной сущности — id пользователя, id тенанта, — а не по объекту: shard = hash(userId) % N → бакет media-3 → его endpoint. Чтение вычисляет тот же hash. Файлы сущности лежат вместе, листинг по префиксу работает, а ребаланс переносит сущности, а не случайные объекты.
  3. Директория в вашей БД. Выбирайте сервер в момент загрузки и храните (key, endpoint) в строке, которую вы и так ведёте для файла. Чтение всё равно идёт через эту строку, так что endpoint достаётся бесплатно; ребаланс — это копирование + обновление строки.

Назначая сервер новой сущности, выбирайте его так же, как Boins выбирает репозитории: случайно, с весом по свободному месту (из /_admin/state, с кэшем на минуты). Presigned-ссылки работают при любой схеме — бэкенд подставляет нужный хост при подписи.

Планирование, чтобы бакет не перерос сервер:

  • Если вид данных может расти неограниченно — шардируйте с первого дня: заведите media-0…media-7, даже если все восемь стартуют на одном сервере. Тогда «разделить» в будущем — это «перевезти целые бакеты» (тривиально, рецепт 12.8), а не «распилить один бакет» (миграция ниже).
  • Ключи всегда несут префикс сущности (users/{id}/…) — именно это делает возможными ленивую миграцию и переезды по-сущностно.
  • Следите за /_admin/state и действуйте при ~70 % заполнения диска: рецепт 12.7 (добавить диск) дёшево покупает время.

Бакет всё-таки перерос сервер — процедура разделения:

  1. Купите время: добавьте репозиторий на новом диске (12.7).
  2. Создайте шард-бакеты на двух (или более) серверах, например media-0 на A, media-1 на B. Новые записи сразу маршрутизируйте по hash сущности.
  3. Старые сущности мигрируйте в фоне, префикс за префиксом: aws s3 sync/rclone каталога users/{id}/… в новый шард сущности.
  4. На время миграции читайте с фолбэком: сначала новое место, при 404 — старый бакет (или ведите в БД флаг «мигрирован» по сущности — так чище).
  5. Когда синхронизация завершена, уберите фолбэк и удалите старый бакет из конфига.

Если в ключах нет префикса сущности и hash невозможен — используйте схему 3: новые загрузки направляйте в новый бакет через строку БД, старые строки оставьте на старом сервере и мигрируйте лениво или вовсе не мигрируйте.

14. Where Boins can go14. Пути развития Boins

Boins 2.0 is deliberately small. These are the directions that fit its philosophy — each adds capability without adding a cluster to babysit. Roughly in order of expected value:

  • Async bucket replication. A built-in follower mode: a second Boins pulls a bucket over the S3 API (ListObjects + conditional GET) on an interval. Warm standby and off-site copies without external tooling — the biggest durability win available without touching the core.
  • Read-time integrity. The MD5 of every blob is already in the index record; an opt-in mode could verify it on read, and a background scrubber could walk repositories and quarantine bit-rot into the fault store.
  • Space reclamation. A garbage collector for blobs orphaned by crash windows between the key index and the core, and a compactor that rewrites sparse blob files (after mass deletions) into dense ones — returning space to the filesystem, not only to Boins.
  • Key index beyond RAM. Today a bucket's key index lives in memory (~100 bytes per object). A tiered mode — hot prefix cache + on-disk sorted runs — would lift the practical per-bucket ceiling into the billions.
  • More S3 surface. Versioning, object tagging, lifecycle/TTL rules (auto-expiry is a natural fit for the delete-and-reuse engine), POST policy uploads, per-bucket quotas.
  • Observability. A Prometheus /metrics endpoint (the collector already holds the numbers), S3 access logs, and a minimal HTML admin page over the existing JSON.
  • The host-selector client. The thin library sketched in recipe 12.11: bucket map + entity rendezvous hashing + free-space weights from /_admin/state — as a tiny boins-client artifact.
  • Performance experiments. Zero-copy GET (FileChannel.transferTo to the socket), group commit for fsync=ALWAYS (amortize the sync across concurrent writers), and io_uring-style backends as the JDK exposes them.

The project is Apache-2.0 — issues and pull requests for any of the above are welcome. The unfashionable goal stays the same: stay small enough to read in an afternoon.

Boins 2.0 намеренно мал. Вот направления, которые вписываются в его философию — каждое добавляет возможности, не добавляя кластера, за которым нужно следить. Примерно в порядке ожидаемой пользы:

  • Асинхронная репликация бакетов. Встроенный режим «ведомого»: второй Boins по интервалу подтягивает бакет через S3 API (ListObjects + условные GET). Тёплый резерв и офсайт-копии без внешних инструментов — самый большой выигрыш в надёжности, доступный без изменений ядра.
  • Целостность при чтении. MD5 каждого блоба уже лежит в индексной записи; опциональный режим мог бы сверять его при чтении, а фоновый «скраббер» — обходить репозитории и отправлять битые блобы в карантин через хранилище исключений.
  • Возврат места. Сборщик мусора для блобов-сирот из крэш-окон между индексом ключей и ядром, и компактор, переписывающий разреженные blob-файлы (после массовых удалений) в плотные — возвращая место файловой системе, а не только Boins.
  • Индекс ключей за пределами RAM. Сегодня индекс ключей бакета живёт в памяти (~100 байт на объект). Ярусный режим — горячий кэш префиксов + сортированные сегменты на диске — поднял бы практический потолок бакета до миллиардов объектов.
  • Больше поверхности S3. Версионирование, теги объектов, lifecycle/TTL-правила (автопротухание естественно ложится на движок удаления-и-переиспользования), POST policy uploads, квоты по бакетам.
  • Наблюдаемость. Prometheus-эндпоинт /metrics (коллектор уже хранит все цифры), access-логи S3 и минимальная HTML-страница админки поверх существующего JSON.
  • Клиент-селектор хостов. Тонкая библиотека из рецепта 12.11: карта бакетов + rendezvous-hash по сущности + веса по свободному месту из /_admin/state — как крошечный артефакт boins-client.
  • Эксперименты с производительностью. Zero-copy GET (FileChannel.transferTo в сокет), групповой коммит для fsync=ALWAYS (амортизация синка между конкурентными писателями), бэкенды в духе io_uring по мере их появления в JDK.
